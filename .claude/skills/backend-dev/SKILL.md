## 开发流程

```
1. 查阅项目配置 → 2. 搜索相关资产 → 3. 阅读相关代码 → 4. 遵循模式开发 → 5. 记录产出
```

### 1. 查阅项目配置

阅读上方「项目配置」章节，了解技术栈和代码模式。

### 2. 搜索相关资产

```bash
project search <关键词>    # 查找已记录的 models/apis/utils
```

### 3. 阅读相关代码

**资产记录不能替代阅读代码。** 找到相关文件后必须阅读理解：

- 用 Grep 搜索相关类/方法
- 用 Read 阅读具体实现
- 复杂情况调用 Explore subagent

### 4. 遵循模式开发

按照探索发现的模式编写代码，保持风格一致。

### 5. 记录产出

```bash
task done T001 '{
  "summary": "做了什么",
  "models": ["Model: 字段说明"],
  "apis": ["/api/path: 接口说明"],
  "utils": ["方法名: 功能说明"]
}'
```

## 通用原则

### SOLID

| 原则 | 实践 |
|-----|------|
| 单一职责 | 一个类/函数只做一件事 |
| 开闭原则 | 扩展开放，修改关闭 |
| 依赖倒置 | 依赖抽象，通过注入获取依赖 |

### 代码质量

- 防御式编程：检查空值、校验输入
- 清晰命名：变量名、函数名要自解释
- 不要硬编码：配置、SQL、提示词都要外置

---

## 禁止事项

- 不要跳过阅读代码直接开发
- 不要假设技术栈和模式
- 不要发明新模式，遵循项目现有风格
- 不要主动写测试（由 /test 触发）
- 不要主动写文档（除非用户要求）
